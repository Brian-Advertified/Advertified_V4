# Architecture Review

## - [x] Executive Summary

Advertified V4 is structurally healthy enough to build, test, and ship across the API and web applications, but it has a few architectural pressure points where convenience code had started to leak across boundaries. The clearest examples in this review pass were duplicated client-side session persistence logic, scattered auth-header construction, small but repeated UI styling exceptions, and resilience gaps around external AI quota failures.

This review focuses on issues that were verified directly in the codebase and then remediated in the same pass so the document can serve as a real working checklist. The result is a tighter auth/session boundary in the web app, more consistent API access patterns, better resilience in the AI integration path, cleaner UI styling for progress components, and a more accurate top-level project guide.

## - [x] Code Organization & Structure

### - [x] Strengths
- [x] The repository has a clear top-level split between `src/Advertified.App` and `src/Advertified.Web`, which keeps backend and frontend concerns physically separated.
- [x] Recent feature extraction work has already reduced some of the worst page-level component sprawl in the web app.

### - [x] Issues
- [x] 🟠 **Duplicated session persistence logic in the web layer** - Session storage and token retrieval were implemented in more than one place, which made auth behavior harder to reason about and increased drift risk between login and app boot flows. Fixed by introducing a shared session store abstraction in `src/Advertified.Web/src/services/sessionStore.ts:21` and consuming it from `src/Advertified.Web/src/features/auth/auth-context.tsx:11` and `src/Advertified.Web/src/services/advertifiedApi.ts:1382`.
- [x] 🟠 **Low-level transport and auth concerns were embedded inside the monolithic API facade** - Error shaping, base-URL handling, downloads, and auth/session flows were mixed into the same module as feature-specific endpoint methods, which made the web data layer harder to evolve safely. Fixed by extracting transport into `src/Advertified.Web/src/services/apiClient.ts:1` and auth/session flows into `src/Advertified.Web/src/services/authApi.ts:1`, with `src/Advertified.Web/src/services/advertifiedApi.ts:55` now acting more as a facade than a catch-all implementation bucket.
- [x] 🟠 **Admin endpoint behavior was still bundled into the remaining web API facade** - A large, tightly related set of admin operations still lived inline in the generic `advertifiedApi` object, which kept the file large and made admin-only changes noisier than necessary. Fixed by extracting the admin feature client into `src/Advertified.Web/src/services/adminApi.ts:1` and composing it into the facade from `src/Advertified.Web/src/services/advertifiedApi.ts:1044`.
- [x] 🟠 **Client campaign and public proposal workflows were still bundled into the remaining web API facade** - Client campaign detail, proposal review, notifications, and campaign approval mutations still lived inline in the generic API service, even though they form a coherent feature slice. Fixed by extracting those workflows into `src/Advertified.Web/src/services/campaignApi.ts:1` and composing them into the facade from `src/Advertified.Web/src/services/advertifiedApi.ts:1049`.
- [x] 🟡 **Workflow-state helpers were duplicated across multiple frontend modules** - Recommendation approval completion, AI-studio readiness, and recommendation selection fallback rules were repeated in several files, creating a correctness risk whenever one flow changed and the others did not. Fixed by centralizing status policy in `src/Advertified.Web/src/lib/campaignStatus.ts:1` and selection logic in `src/Advertified.Web/src/features/campaigns/recommendationSelection.ts:1`, then consuming those helpers from `src/Advertified.Web/src/lib/access.ts:1`, `src/Advertified.Web/src/pages/client/CampaignDetailPage.tsx:1`, `src/Advertified.Web/src/pages/public/ProposalEntryPage.tsx:1`, `src/Advertified.Web/src/pages/client/clientWorkspace.tsx:1`, and `src/Advertified.Web/src/features/campaigns/aiStudioAccess.ts:1`.
- [x] 🟡 **Missing top-level onboarding documentation** - The repository lacked a root-level architectural README, which made local setup and system boundaries harder to understand from the repo entry point. Fixed by adding `README.md:1`.
- [x] 🟢 **Temporary generated artifact was not ignored** - The generated proposal PDF temp artifact could pollute the working tree and distract code review. Fixed in `.gitignore:35`.

- [x] 🟠 **Agent workflow endpoints were still bundled into the remaining web API facade** - Agent campaign detail, inbox, messaging, recommendation, inventory, and prospect flows were still mixed into the generic facade, which kept one of the busiest operational areas coupled to unrelated features. Fixed by extracting the agent feature client into `src/Advertified.Web/src/services/agentApi.ts:1` and composing it into the facade from `src/Advertified.Web/src/services/advertifiedApi.ts:1001`.
- [x] 🟠 **AI platform endpoint behavior was still bundled into the remaining web API facade** - AI studio jobs, asset generation, QA, voice selection, and ad-ops methods still lived inline in `advertifiedApi`, which made the file a cross-feature hotspot and raised the risk of provider-facing changes bleeding into unrelated areas. Fixed by extracting those workflows into `src/Advertified.Web/src/services/aiPlatformApi.ts:1` and composing them into the facade from `src/Advertified.Web/src/services/advertifiedApi.ts:1002`.
- [x] 🟠 **Creative workflow endpoints were still bundled into the remaining web API facade** - Creative inbox, campaign detail, asset upload, approval handoff, and creative-system generation were still living inline in the generic facade, which kept creative-ops changes mixed with unrelated customer and AI workflows. Fixed by extracting the creative feature client into `src/Advertified.Web/src/services/creativeApi.ts:1` and composing it into the facade from `src/Advertified.Web/src/services/advertifiedApi.ts:1002`.

- [x] 🟡 **Agent data-layer behavior was not isolated from unrelated client, admin, and AI endpoints** - Agent inbox, campaign, messaging, recommendation, inventory, and prospect operations were still mixed into the generic facade, which made the highest-churn operational workflows harder to reason about. Fixed by moving those endpoints into `src/Advertified.Web/src/services/agentApi.ts:1`.
- [x] 🟡 **AI platform data-layer behavior was not isolated from unrelated client, admin, and agent endpoints** - AI studio job and asset orchestration paths were still mixed into the generic facade, which made provider-facing changes noisy and risky. Fixed by moving those endpoints into `src/Advertified.Web/src/services/aiPlatformApi.ts:1`.
- [x] 🟡 **Creative data-layer behavior was not isolated from unrelated client, agent, and AI endpoints** - Creative inbox, asset upload, approval handoff, and creative-system generation were still mixed into the generic facade, which made studio-side changes harder to isolate and review. Fixed by moving those endpoints into `src/Advertified.Web/src/services/creativeApi.ts:1`.

### - [x] Recommendations
- [x] Keep client-side session persistence behind `sessionStore.ts` and treat direct storage access elsewhere as a regression.
- [x] Continue splitting `advertifiedApi.ts` by responsibility, but keep `apiClient.ts` as the single place for request, download, and API error behavior.
- [x] Prefer feature-scoped service modules like `adminApi.ts` for large endpoint clusters instead of regrowing a single global API bucket.
- [x] Group client-facing campaign, proposal, and notification workflows behind one feature client so approval-path changes stay localized.
- [x] Keep agent campaign, inbox, messaging, recommendation, and inventory workflows behind one feature client so operational changes stay localized.
- [x] Keep AI platform jobs, assets, QA, voice, and ad-ops workflows behind a dedicated feature client so provider-facing changes remain isolated from the rest of the app.
- [x] Keep creative inbox, campaign, asset, and approval workflows behind one feature client so studio-side changes stay isolated.
- [x] Keep packages, checkout, orders, partner enquiry, and consent flows behind one public-facing feature client so customer-commerce changes stay isolated from internal tooling.
- [x] Keep campaign-workflow state rules in shared policy modules instead of encoding status arrays inline in page components.
- [x] Extend the new root `README.md` with deployment and troubleshooting links as adjacent architecture changes land.

## - [x] State Management

### - [x] Strengths
- [x] The web app distinguishes server state from UI state reasonably well, with API interactions centralized through the `advertifiedApi` service.
- [x] The new shared session store makes auth state bootstrapping and persistence more predictable.

### - [x] Issues
- [x] 🟠 **Auth state bootstrapping was split across multiple storage implementations** - `auth-context` and `advertifiedApi` each maintained their own session persistence rules, which made it harder to trace state flow and increased the chance of inconsistent logout/login behavior. Fixed through the shared read/write path in `src/Advertified.Web/src/services/sessionStore.ts:21` and its adoption in `src/Advertified.Web/src/features/auth/auth-context.tsx:36` and `src/Advertified.Web/src/services/advertifiedApi.ts:1524`.
- [x] 🟡 **Legacy session migration path was implicit rather than explicit** - Existing `localStorage` sessions needed a controlled migration path once storage rules changed. Fixed with legacy-to-current migration logic in `src/Advertified.Web/src/services/sessionStore.ts:28`.
- [x] 🟡 **Recommendation selection fallback order was duplicated across client and public approval flows** - Both approval experiences separately implemented “current, approved, requested, sent-to-client, first” selection logic, which invited subtle divergence. Fixed by centralizing the behavior in `src/Advertified.Web/src/features/campaigns/recommendationSelection.ts:1` and reusing it from `src/Advertified.Web/src/pages/client/CampaignDetailPage.tsx:1` and `src/Advertified.Web/src/pages/public/ProposalEntryPage.tsx:1`.

### - [x] Recommendations
- [x] Route all future browser-session reads and writes through `readStoredSession` and `writeStoredSession`.
- [x] Add tests whenever auth state shape changes so bootstrapping, logout, and token-header behavior remain traceable.
- [x] Reuse the shared recommendation-selection helpers whenever a screen needs to choose the “active” proposal from campaign data.

## - [x] Security

### - [x] Strengths
- [x] Route protection is centralized through the web app’s protected-route layer, which gives the application a single guard entry point.
- [x] Backend identity and token handling remain server-side concerns rather than being recreated ad hoc in multiple backend services.

### - [x] Issues
- [x] 🔴 **Session tokens were persisted in `localStorage`** - Persisting bearer tokens in `localStorage` increased exposure to token theft in the event of XSS and made session lifetime harder to constrain. Fixed by moving active session persistence to `sessionStorage` in `src/Advertified.Web/src/services/sessionStore.ts:5` and removing duplicated direct storage handling from `src/Advertified.Web/src/features/auth/auth-context.tsx:36` and `src/Advertified.Web/src/services/advertifiedApi.ts:1524`.
- [x] 🟠 **Auth header construction was duplicated in upload code paths** - Several upload flows manually rebuilt bearer headers, increasing the chance of drift or missing fixes in future security changes. Fixed by centralizing header generation in `src/Advertified.Web/src/services/sessionStore.ts:64` and consuming it from `src/Advertified.Web/src/services/advertifiedApi.ts:1736`, `src/Advertified.Web/src/services/advertifiedApi.ts:2363`, and `src/Advertified.Web/src/services/advertifiedApi.ts:2381`.

### - [x] Recommendations
- [x] Treat `sessionStorage` plus centralized header construction as the only approved browser auth-token pattern.
- [x] If the platform moves to cookie-based auth later, replace the shared session-store implementation once rather than editing each API callsite.

- [x] The progress components now share a single styling hook for width presentation.

### - [x] Issues

Fixed by moving the width contract to a CSS custom property in `src/Advertified.Web/src/index.css:569` and updating `src/Advertified.Web/src/components/campaign/CampaignStepper.tsx:107` and `src/Advertified.Web/src/components/agent/AgentStepper.tsx:87`.

### - [x] Recommendations
- [x] Use component-level CSS variables for dynamic visual values before falling back to ad hoc inline style objects.
- [x] Keep shared visual primitives in `index.css` or a dedicated shared stylesheet so similar widgets stay aligned.

## - [x] Performance

### - [x] Strengths
- [x] The app already uses modern build tooling and route-based chunking, which keeps the platform shippable at its current scale.
- [x] Recent planner work has reduced failure amplification around external AI calls.

### - [x] Issues
- [x] 🟠 **External AI quota failures could trigger repeated wasted calls** - OpenAI `429 insufficient_quota` responses were not previously being treated as a distinct resilience condition, which increased latency and log noise during degraded periods. Fixed with quota backoff handling in `src/Advertified.App/Services/OpenAICampaignReasoningService.cs:25` and no-retry behavior for OpenAI `429` responses in `src/Advertified.App/Support/TransientHttpRetryHandler.cs:53`.
- [x] 🟠 **Planner candidate loading had a concurrency hazard against the EF data path** - Concurrent candidate resolution created correctness and reliability risk in the recommendation generation path. Fixed in `src/Advertified.App/Services/PlanningCandidateLoader.cs:1`.

### - [x] Recommendations
- [x] Keep vendor-specific retry policy logic close to the HTTP client infrastructure instead of scattering it in feature code.
- [x] Prefer deterministic fallback behavior over retry storms when external AI providers are unavailable or out of quota.

## - [x] Error Handling & Resilience

### - [x] Strengths
- [x] The API already has centralized exception middleware, which gives the backend a consistent place to evolve error shaping.
- [x] The AI recommendation path now degrades more gracefully when quota is exhausted.

### - [x] Issues
- [x] 🟠 **Quota exhaustion did not fail fast enough for dependent recommendation flows** - Downstream recommendation generation could spend unnecessary time waiting on external calls that had no chance of succeeding under active quota exhaustion. Fixed through backoff-aware short-circuiting in `src/Advertified.App/Services/OpenAICampaignReasoningService.cs:53`.
- [x] 🟠 **Recommendation approval relied on string-matched exceptions across service, controller, and middleware layers** - Approval and public proposal flows depended on exact `InvalidOperationException` messages for HTTP behavior, which was brittle and easy to break during copy or wording changes. Fixed with typed `BadRequestException` and `PaymentRequiredException` contracts in `src/Advertified.App/Support/HttpExceptions.cs:6`, mapped centrally in `src/Advertified.App/Middleware/ProblemDetailsExceptionHandlingMiddleware.cs:54`, and consumed from `src/Advertified.App/Services/RecommendationApprovalWorkflowService.cs:54`, `src/Advertified.App/Controllers/CampaignsController.cs:192`, and `src/Advertified.App/Controllers/PublicProposalController.cs:64`.
- [x] 🟡 **Async loading and error states were implemented with repeated bespoke query guards** - Agent, admin, and creative entry pages each owned their own loading/error shell logic, which made empty/error handling drift likely over time. Fixed with a shared query boundary in `src/Advertified.Web/src/components/ui/QueryStateBoundary.tsx:5`, adopted by `src/Advertified.Web/src/pages/agent/agentWorkspace.tsx:192`, `src/Advertified.Web/src/pages/admin/adminWorkspace.tsx:179`, and `src/Advertified.Web/src/pages/creative/CreativeDirectorPages.tsx:90`.

### - [x] Recommendations
- [x] Preserve the current “log once, back off, continue with fallback behavior” pattern for other third-party dependencies.
- [x] Add integration tests for degraded external-provider states when new AI-dependent workflows are introduced.
- [x] Prefer typed HTTP-facing exceptions over message parsing whenever backend services need to signal request semantics.
- [x] Reuse the shared query boundary for future workspace-style pages instead of hand-rolling per-page loading/error wrappers.

### - [x] Strengths

- [x] The backend’s retry and provider logic are now better isolated from feature flow code.

### - [x] Issues
- [x] 🟠 **Authenticated request construction was inconsistent inside the central API client** - Even with a single API service file, some authenticated requests still bypassed shared header logic and rebuilt transport concerns manually. Fixed by consolidating those paths onto `buildAuthenticatedHeaders` in `src/Advertified.Web/src/services/sessionStore.ts:64` and reusing it from `src/Advertified.Web/src/services/advertifiedApi.ts:1382`.

Fixed with `BadRequestException` and `PaymentRequiredException` in `src/Advertified.App/Support/HttpExceptions.cs:6` and middleware coverage in `tests/Advertified.App.Tests/UnitTest1.cs:92`.
- [x] 🟡 **Web transport behavior was not isolated from feature endpoint definitions** - The API facade carried its own request, download, and error-handling machinery inline, which made data-layer fixes harder to share across features. Fixed by extracting the shared transport layer into `src/Advertified.Web/src/services/apiClient.ts:1`.
- [x] 🟡 **Admin data-layer behavior was not isolated from unrelated client, campaign, and AI endpoints** - Keeping admin endpoints inline with every other feature path increased cognitive load and made role-specific changes harder to review. Fixed by moving those endpoints into `src/Advertified.Web/src/services/adminApi.ts:1`.
- [x] 🟡 **Campaign and proposal data-layer behavior was not isolated from unrelated admin, agent, and AI endpoints** - Client approval and proposal workflows were mixed into the same facade as unrelated feature areas, which made the data layer harder to navigate and reason about. Fixed by moving those endpoints into `src/Advertified.Web/src/services/campaignApi.ts:1`.

### - [x] Recommendations
- [x] Continue collapsing repeated fetch-header and session-token code into reusable transport helpers before splitting the large API client file further.
- [x] When `advertifiedApi.ts` is broken into feature clients, move shared auth/header logic intact rather than duplicating it by feature.
