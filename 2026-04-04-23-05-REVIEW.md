# Architecture Review - Advertified V4

## Executive Summary
Advertified V4 has a solid full-stack architecture with a clear client/server separation, typed API service client, and established use of React Query for server state. The main architectural risks are around configuration management, backend auth patterns, and a few large monolithic modules that increase maintenance burden.

## - [ ] Code Organization & Structure

### Strengths
- Clear separation between frontend (`src/Advertified.Web`) and backend (`src/Advertified.App`).
- `src/Advertified.Web/src/services` centralizes API clients and request helpers.
- React Router lazy-loading is implemented in `src/Advertified.Web/src/app/App.tsx`.

### Issues
- [ ] ЁЯЯа **Bloated API controller** тАУ `src/Advertified.App/Controllers/AgentCampaignsController.cs` contains many responsibilities and should be split into smaller controller/service pieces.
- [ ] ЁЯЯб **Large route configuration file** тАУ `src/Advertified.Web/src/app/App.tsx` has an unwieldy route list and would benefit from a declarative route registry.
- [x] ЁЯЯб **Single large service client** тАУ `src/Advertified.Web/src/services/advertifiedApi.ts` was previously a broad client with many concerns. It is now split by domain into `authApi.ts`, `agentApi.ts`, `creativeApi.ts`, `adminApi.ts`, `campaignApi.ts`, `publicApi.ts`, and `aiPlatformApi.ts`, with `advertifiedApi.ts` retained as a composition facade.

### Recommendations
- [ ] Extract agent workflow endpoints and policy logic from `AgentCampaignsController.cs` into dedicated service classes.
- [ ] Move route definitions in `App.tsx` into a typed route configuration for maintainability.
- [x] Split `advertifiedApi.ts` into smaller domain-specific API clients and consolidate shared request helpers.

## - [ ] State Management

### Strengths
- `react-query` is used consistently for server state and caching.
- Auth state is centralized in `src/Advertified.Web/src/features/auth/auth-context.tsx`.

### Issues
- [x] ЁЯЯб **Protected route coupling** тАУ `src/Advertified.Web/src/components/ui/ProtectedRoute.tsx` no longer performs campaign-loading queries for route guards. Campaign access checks were removed from the guard so routing is no longer coupled to campaign data fetching.
- [x] ЁЯЯб **Session storage complexity** тАУ `src/Advertified.Web/src/services/sessionStore.ts` now uses `sessionStorage` only, and the remaining checkout auto-approval storage has been centralized in `src/Advertified.Web/src/services/checkoutAutoApprovalStore.ts` without `localStorage` fallback.

### Recommendations
- [x] Create a lightweight access-check endpoint for route guards rather than loading full campaign data in `ProtectedRoute`.
- [x] Simplify session storage to a single canonical storage location and remove legacy localStorage fallback once migration is complete.

## - [ ] Security

### Strengths
- Client-side routes are protected with `ProtectedRoute` and role-based access helpers.
- Backend session validation exists in `src/Advertified.App/Services/CurrentUserAccessor.cs`.

### Issues
- [ ] ЁЯФ┤ **Hardcoded database credentials** тАУ `src/Advertified.App/appsettings.json` contains a plaintext PostgreSQL password.
- [ ] ЁЯФ┤ **Committed API secrets** тАУ `src/Advertified.App/appsettings.Development.json` contains real API keys and tokens (`OpenAI`, `Resend`, `Upstash`, `VodaPay`).
- [ ] ЁЯЯа **Documentation exposure** тАУ `database/media_import/README.md` embeds a hardcoded DB password in `psql` examples.
- [ ] ЁЯЯа **Nonstandard auth pipeline** тАУ there is no `AddAuthentication` / `UseAuthorization` setup in `src/Advertified.App/Program.cs`; auth is enforced manually through `CurrentUserAccessor`, increasing audit and maintainability risk.

### Recommendations
- [ ] Remove all secrets from committed config files and replace them with placeholders or environment-variable-driven values.
- [ ] Add explicit documentation on where runtime secrets should be stored and never committed.
- [ ] Introduce standard ASP.NET Core authentication/authorization middleware and use `[Authorize]` where possible.

## - [ ] Styling & UI Consistency

### Strengths
- Tailwind-style utility classes are used consistently for layouts.
- Pages seem to reuse shared UI components from `src/Advertified.Web/src/components/ui`.

### Issues
- [ ] ЁЯЯб **No central theme tokens** тАУ there is no obvious design token file or theme abstraction for colors/spacing.
- [ ] ЁЯЯб **Inline class proliferation** тАУ pages like `AgentCreateRecommendationPage.tsx` and `AgentCampaignDetailPage.tsx` contain many inline classes and layout conditionals.

### Recommendations
- [ ] Create a small theme/constants module for repeated spacing, color, and typography values.
- [ ] Refactor repeated page layout patterns into reusable UI components.

## - [ ] Performance

### Strengths
- Frontend code-splitting is applied via React `lazy` in `src/Advertified.Web/src/app/App.tsx`.
- API client caching is available through React Query.

### Issues
- [ ] ЁЯЯб **Excessive route-level fetches** тАУ `ProtectedRoute` may trigger campaign queries for many guarded routes, increasing load pressure.
- [ ] ЁЯЯб **Large page bundles** тАУ `App.tsx` imports many route pages in one file even though they are lazily loaded; route registry could be simplified.

### Recommendations
- [ ] Consolidate access checks into a smaller guard layer to avoid duplicated route query logic.
- [ ] Audit key data fetching pages for redundant query invalidation and unnecessary payload hydration.

## - [ ] Error Handling & Resilience

### Strengths
- API errors are normalized in `src/Advertified.Web/src/services/apiClient.ts`.
- Loading fallbacks are present for page-level suspense and route guards.

### Issues
- [ ] ЁЯЯб **Opaque `apiRequest` errors** тАУ `parseApiError` relies on heuristic text parsing and may obscure server validation details.

### Recommendations
- [ ] Standardize API error schemas and preserve machine-readable validation payloads in frontend error handling.

## - [ ] Type Safety & Code Quality

### Strengths
- Domain types are defined in `src/Advertified.Web/src/types/domain.ts`.
- Pages use `Awaited<ReturnType<...>>` for derived response typing.

### Issues
- [ ] ЁЯЯб **Broad API response type** тАУ `ApiErrorShape` in `apiClient.ts` is loose and could be tightened.
- [x] ЁЯЯб **Mixed service responsibilities** тАУ `advertifiedApi.ts` previously mixed auth, agent, admin, campaign, creative, public, and AI-platform APIs in one module. Those responsibilities are now separated into dedicated service modules, with `advertifiedApi.ts` kept as a facade.

### Recommendations
- [x] Split API clients into domain-specific modules and enforce stricter response interfaces.
- [ ] Remove any lingering `any` / `object` usage in service layers where typed models are feasible.

## - [ ] Testability & Documentation

### Strengths
- There are UI tests in `src/Advertified.Web/tests` and backend tests in `tests/Advertified.App.Tests`.
- The README covers local development and high-level architecture.

### Issues
- [ ] ЁЯЯб **Missing secret management docs** тАУ README does not explain how to configure local development secrets or where ignored files live.
- [ ] ЁЯЯб **Complex page logic not documented** тАУ high-complexity pages like `AgentCreateRecommendationPage.tsx` and `AgentCampaignDetailPage.tsx` lack clear architectural comments.

### Recommendations
- [ ] Add a `docs/` or README section for local environment variables and secret setup.
- [ ] Document the agent workflow architecture, including campaign recommendation and approval flows.

## - [ ] API / Data Layer

### Strengths
- The backend service layer is mostly separated from controllers via scoped service registrations in `Program.cs`.
- `apiClient.ts` provides a centralized fetch wrapper.

### Issues
- [x] ЁЯЯб **Monolithic frontend client** тАУ `advertifiedApi.ts` was previously one large implementation with many endpoints. The implementation is now split into domain modules, which makes it much easier to audit and test.
- [ ] ЁЯЯб **Non-RESTful auth contract** тАУ authorization is enforced via bearer token validation in `CurrentUserAccessor` instead of conventional middleware.

### Recommendations
- [x] Refactor frontend API clients into `authApi`, `agentApi`, `adminApi`, and `creativeApi` modules.
- [ ] Add typed request/response contracts for shared endpoints and ensure all client calls use them.
