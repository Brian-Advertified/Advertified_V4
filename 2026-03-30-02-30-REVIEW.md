## Executive Summary
- [ ] Overall health is mixed: the codebase has a solid amount of domain modeling, a centralized TypeScript API client, useful backend tests, and clear product momentum across client, agent, admin, and creative workflows. The team has also started moving toward better server-state handling with React Query and shared access helpers on the web.
- [ ] The biggest risks are architectural rather than cosmetic. Backend identity is currently caller-controlled, several operations endpoints rely on frontend route guards instead of hard backend authorization, and password storage uses a fast unsalted hash. Outside security, the main maintainability pressure comes from very large page modules, stringly-typed workflow state, and documentation/tests that lag the current implementation.

## - [ ] Code Organization & Structure

### ✅ Strengths
- [ ] The repository has a recognizable split between `src/Advertified.App`, `src/Advertified.Web`, `database`, and `tests`, which gives backend, frontend, schema, and validation work clear homes.
- [ ] The backend has meaningful service and contract layers instead of collapsing all logic into controllers, for example the separation between controllers and services under `src/Advertified.App/Controllers` and `src/Advertified.App/Services`.
- [ ] No obvious circular dependency hotspots were found from import/controller inspection; the main problem is module size, not cycle churn.

### ⚠️ Issues
- [ ] 🟠 **Admin UI is concentrated into a monolith** – `src/Advertified.Web/src/pages/admin/AdminSectionPages.tsx:1` contains many unrelated admin screens and mutations in one 1,800+ line file, which makes it difficult to reason about ownership, reuse, and safe edits.
- [ ] 🟠 **Client and agent campaign screens are overloaded** – `src/Advertified.Web/src/pages/client/CampaignDetailPage.tsx:1`, `src/Advertified.Web/src/pages/client/clientWorkspace.tsx:1`, and `src/Advertified.Web/src/pages/agent/AgentCampaignDetailPage.tsx:1` each mix data fetching, workflow rules, rendering, and mutation orchestration in single modules.
- [ ] 🟡 **Repeated mutation invalidation patterns increase drift risk** – the same query invalidation bundle is repeated across multiple agent mutations in `src/Advertified.Web/src/pages/agent/AgentCampaignDetailPage.tsx:283` and again in other pages such as `src/Advertified.Web/src/pages/client/CampaignDetailPage.tsx:23`.
- [ ] 🟡 **Documentation does not describe the actual product** – `src/Advertified.Web/README.md:1` is still the stock Vite template and does not document current routing, auth assumptions, workflows, or build/runtime requirements.

### 🔧 Recommendations
- [ ] Break `AdminSectionPages.tsx` into route-level screen modules plus smaller shared admin feature components by domain: outlets, pricing, geography, users, imports, audit.
- [ ] Extract campaign workflow orchestration into dedicated hooks/helpers so the page files mostly render UI and invoke named operations.
- [ ] Introduce a shared query-key and invalidation helper layer for campaign, inbox, and admin dashboard refreshes.
- [ ] Replace the template README with project-specific setup, environment, architecture, and workflow documentation.

## - [ ] State Management

### ✅ Strengths
- [ ] React Query is being used consistently for most server state on the web, for example in `src/Advertified.Web/src/pages/public/PackagesPage.tsx:19`, `src/Advertified.Web/src/pages/client/CampaignDetailPage.tsx:18`, and `src/Advertified.Web/src/pages/agent/AgentMessagesNotesPage.tsx:19`.
- [ ] Shared workflow helpers such as `getCampaignProgressPercent` and `getCampaignQuickSteps` in `src/Advertified.Web/src/pages/client/clientWorkspace.tsx:8` help keep client progress logic in one place.

### ⚠️ Issues
- [ ] 🟠 **Auth state is split across module-level storage and the API layer** – `src/Advertified.Web/src/features/auth/auth-context.tsx:21` keeps session state in module globals while `src/Advertified.Web/src/services/advertifiedApi.ts:415` and `src/Advertified.Web/src/services/advertifiedApi.ts:764` independently read from local storage, which makes session flow harder to audit and replace safely.
- [ ] 🟠 **Global notification state pulls live data directly from multiple domains** – `src/Advertified.Web/src/features/notifications/NotificationCenter.tsx:224` issues route-independent queries for campaigns, orders, agent inbox, and admin dashboard from the navbar layer, increasing cross-cutting fetch load and coupling.
- [ ] 🟡 **State transitions are still string-driven across layers** – workflow state is encoded as hardcoded strings in both backend and frontend, for example `src/Advertified.App/Controllers/CampaignsController.cs:184` and `src/Advertified.Web/src/pages/client/clientWorkspace.tsx:16`, which raises regression risk as the lifecycle grows.

### 🔧 Recommendations
- [ ] Move session read/write responsibilities behind one auth/session abstraction and stop letting the generic API client reach into browser storage directly.
- [ ] Convert notification data into a smaller server-side summary endpoint or a dedicated aggregated query per role instead of fan-out querying from the shell.
- [ ] Promote campaign statuses and transition rules into centralized constants/enums shared per layer rather than repeated string literals.

## - [ ] Security

### ✅ Strengths
- [ ] Some endpoints do perform resource ownership checks, for example client campaign retrieval in `src/Advertified.App/Controllers/CampaignsController.cs:64` and admin-only gates in `src/Advertified.App/Controllers/AdminController.cs:847`.
- [ ] Registration and campaign brief requests already use validation classes under `src/Advertified.App/Validation`.

### ⚠️ Issues
- [ ] 🔴 **Backend identity is spoofable and unauthenticated** – the frontend sends caller identity through `X-User-Id` in `src/Advertified.Web/src/services/advertifiedApi.ts:764`, the server trusts that header in `src/Advertified.App/Services/CurrentUserAccessor.cs:21`, and `src/Advertified.App/Program.cs:186` has no authentication/authorization middleware at all.
- [ ] 🔴 **Missing identity falls back to the most recently created user** – `src/Advertified.App/Services/CurrentUserAccessor.cs:27` silently resolves requests as the newest account when no user header is present, which can expose data or mutate the wrong account.
- [ ] 🔴 **Passwords are stored and checked with unsalted SHA-256** – login hashes directly in `src/Advertified.App/Controllers/AuthController.cs:51` and `src/Advertified.App/Controllers/AuthController.cs:107`, while registration/admin creation also write SHA-256 hashes in `src/Advertified.App/Services/RegistrationService.cs:50` and `src/Advertified.App/Controllers/AdminController.cs:1018`.
- [ ] 🔴 **Operations endpoints rely on the frontend for authorization** – agent endpoints return or mutate all campaigns without role gates in `src/Advertified.App/Controllers/AgentCampaignsController.cs:57`, `src/Advertified.App/Controllers/AgentCampaignsController.cs:174`, and `src/Advertified.App/Controllers/AgentCampaignsController.cs:204`, while creative send-back actions also lack a role gate in `src/Advertified.App/Controllers/CreativeCampaignsController.cs:24`.
- [ ] 🟠 **Agent messaging is not scoped to assignment or ownership** – the inbox enumerates every campaign in `src/Advertified.App/Controllers/CampaignMessagingController.cs:42`, and thread access loads any campaign by id in `src/Advertified.App/Controllers/CampaignMessagingController.cs:236`, so any authenticated operations user could read or send messages on unrelated campaigns.
- [ ] 🟠 **Frontend route guards create a false sense of protection** – `src/Advertified.Web/src/components/ui/ProtectedRoute.tsx:46` blocks unauthorized navigation in the browser, but backend enforcement is inconsistent, so the browser guard is not a security boundary.

### 🔧 Recommendations
- [ ] Replace `X-User-Id` with a signed server-issued session token and enforce authentication in the API pipeline.
- [ ] Remove the “latest user” fallback entirely and fail closed when no valid session is present.
- [ ] Switch password storage and verification to a slow salted hasher such as ASP.NET Core `PasswordHasher<TUser>` or another KDF-backed mechanism.
- [ ] Add backend role/ownership checks to every agent, creative, and messaging endpoint, including assignment-based access where product rules require it.

### 🔧 Recommendations

- [ ] Introduce reusable shell/layout primitives for role workspaces where the structural patterns are the same.

## - [ ] Performance

### ✅ Strengths
- [ ] There is at least one targeted lazy-load for the preview map in `src/Advertified.Web/src/features/packages/components/SpendPreviewPanel.tsx:6`, which shows the team is already using route/component splitting where it matters.
- [ ] React Query `staleTime` and `refetchOnWindowFocus` defaults in `src/Advertified.Web/src/main.tsx:11` reduce some unnecessary refetch churn.

### ⚠️ Issues
- [ ] 🟠 **Most screens are eagerly loaded through the top-level router** – `src/Advertified.Web/src/app/App.tsx:1` imports all public, client, admin, agent, and creative pages eagerly, which will inflate the initial bundle as the app grows.
- [ ] 🟠 **Operations controllers pull broad datasets and sort/filter in memory** – `src/Advertified.App/Controllers/AgentCampaignsController.cs:60` and `src/Advertified.App/Controllers/CampaignMessagingController.cs:42` load full campaign collections with multiple includes before shaping responses, which will not scale cleanly.
- [ ] 🟡 **Navbar notifications can trigger redundant network work** – `src/Advertified.Web/src/features/notifications/NotificationCenter.tsx:224` fetches multiple large datasets just to produce six visible notification items.

### 🔧 Recommendations
- [ ] Introduce route-level lazy loading for admin, agent, creative, and heavier client pages.
- [ ] Push sorting/filtering/projection closer to SQL with narrower queries and DTO projections for inbox/list endpoints.
- [ ] Replace notification fan-out with role-specific lightweight summaries from the backend.

## - [ ] Error Handling & Resilience

### ✅ Strengths
- [ ] Many pages do provide loading and empty states, for example `src/Advertified.Web/src/pages/client/CampaignDetailPage.tsx:114` and `src/Advertified.Web/src/pages/agent/AgentMessagesNotesPage.tsx:44`.
- [ ] Several backend email operations log and continue instead of hard failing campaign workflows, for example `src/Advertified.App/Controllers/CampaignsController.cs:361` and `src/Advertified.App/Controllers/CampaignMessagingController.cs:381`.

### ⚠️ Issues
- [ ] 🟠 **There is no React error boundary around the application shell** – `src/Advertified.Web/src/main.tsx:20` mounts the full app tree directly, so unexpected render failures will still crash the current route without a controlled fallback.
- [ ] 🟡 **Some pages still lack explicit error states for primary queries** – `src/Advertified.Web/src/pages/public/PackagesPage.tsx:76` handles loading but does not show a first-class error UI for package or preview query failures.
- [ ] 🟡 **Controller exception handling is inconsistent** – some controllers translate domain failures into `ProblemDetails`, while others throw raw `InvalidOperationException` and rely on default handling, for example `src/Advertified.App/Controllers/AgentCampaignsController.cs:186` and `src/Advertified.App/Controllers/CampaignMessagingController.cs:82`.

### 🔧 Recommendations
- [ ] Add a top-level React error boundary with a recoverable fallback and telemetry hook.
- [ ] Standardize query state handling in a small set of reusable boundary components across public, client, agent, and admin pages.
- [ ] Add a global API exception mapping layer so controller/domain failures produce consistent `ProblemDetails` responses.

## - [ ] Type Safety & Code Quality

### ✅ Strengths
- [ ] The TypeScript domain layer is broad and explicit, especially in `src/Advertified.Web/src/types/domain.ts`.
- [ ] Backend request validation is present for several high-value flows, which reduces blind deserialization risk.

### ⚠️ Issues
- [ ] 🟠 **Workflow and role logic depend heavily on magic strings** – status and role comparisons are repeated throughout `src/Advertified.App/Controllers/CampaignsController.cs:184`, `src/Advertified.App/Controllers/AgentCampaignsController.cs:548`, and `src/Advertified.Web/src/pages/client/clientWorkspace.tsx:16`.
- [ ] 🟡 **Unstructured metadata weakens the domain model** – planning and audit paths rely on `Dictionary<string, object?>` and `object?` metadata in `src/Advertified.App/Services/PlanningModels.cs:55` and `src/Advertified.App/Services/ChangeAuditService.cs:26`, which makes refactors and validation harder.
- [ ] 🟡 **Naming consistency is mixed across APIs** – some admin responses expose Pascal-like status strings while frontend code normalizes snake/lower case in multiple places, increasing mapping friction.

### 🔧 Recommendations
- [ ] Consolidate workflow/status constants into reusable per-layer definitions and use transition helpers instead of scattered string comparisons.
- [ ] Replace untyped metadata bags with purpose-specific DTOs where the shape is known and stable.
- [ ] Normalize server/client naming conventions so one layer does not need to repeatedly “fix up” the other.

## - [ ] API / Data Layer

### ✅ Strengths
- [ ] The frontend does have a centralized API abstraction in `src/Advertified.Web/src/services/advertifiedApi.ts`, which is the right place for transport concerns.
- [ ] Backend contracts are explicit and typed across many domains under `src/Advertified.App/Contracts`.

### ⚠️ Issues
- [ ] 🟠 **The API client is centralized but not fully encapsulated** – it still reaches into browser storage and caller-supplied identity inside `src/Advertified.Web/src/services/advertifiedApi.ts:415` and `src/Advertified.Web/src/services/advertifiedApi.ts:757`, so transport, session, and domain concerns are bleeding together.
- [ ] 🟠 **A few backend links and response assumptions lag the current routes** – for example recommendation-ready and message notification flows still construct route-specific URLs in backend controllers such as `src/Advertified.App/Controllers/AgentCampaignsController.cs:885` and `src/Advertified.App/Controllers/CampaignMessagingController.cs:130`, while the client experience has recently been consolidated.
- [ ] 🟡 **Some list endpoints expose broad aggregates where role-targeted DTOs would be healthier** – `src/Advertified.App/Controllers/AgentCampaignsController.cs:57` and `src/Advertified.App/Controllers/CampaignMessagingController.cs:32` return more campaign surface area than the listing views strictly need.

### 🔧 Recommendations
- [ ] Split session/auth transport concerns out of the generic API client into a dedicated session-aware transport layer.
- [ ] Audit all server-generated URLs against the currently active frontend routes and keep them behind one route builder.
- [ ] Prefer view-specific DTO projections for list endpoints instead of hydrating whole entity graphs.
